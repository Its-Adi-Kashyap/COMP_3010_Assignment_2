Part 1 API Webserver - 3010-eeter

Create a multi-threaded web server. Web server must be able to find files on a file system, and serve those files to a HTTP client.

Paths that are prefixes with /api/ are API calls that run code. All other paths are searched on a local filesystem for files. If the files or API paths do not exist, or have problems, appropriate error messages should be returned. I have provided some test web content to serve. This is also how you should be serving your website to show Tweets.

Your application should not crash, and always provide a response, even if it is just an error code.

You may store your tweets on the server however you would like, but the data must be available from all threads on the system, and be persistent through restarts.

Create a web page that is used to interact with your Twitter system. All users can create tweets, and view all tweets. Your page should not refresh at any point. All communication should be done with XHR. Hint: don’t use “on submit” features on forms - they refresh the page.

We will test your site on Google Chrome.

API paths

  • GET /api/tweet - get a list of all memos
  • POST /api/tweet - create a new memo on the server. This message will have a body that is json.
  • POST /api/login - Log into the system, return a 200, and sets cookies
  • DELETE /api/login - Log out of the system, remove the cookie
  • DELETE /api/tweet/<id> - Delete a tweet. You can only delete your own tweets. You should have a ‘delete’ button on the rows of your tweets only. Be sure to do the check of if you can delete a post in the back-end.

Only users with cookies should be able to do anything with tweets (view, post, or delete).
